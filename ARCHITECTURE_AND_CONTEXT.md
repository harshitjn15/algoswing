# AlgoSwing: Project Context & Architecture Document

## 1. Project Overview
**AlgoSwing** is a production-grade, modular monolith algorithmic trading platform and terminal. It is designed to be a highly professional, broker-agnostic system (comparable to Zerodha Kite or Dhan) that handles automated market scanning, strategy-based signal generation, paper/live trading execution, and advanced charting.

**Key Philosophies:**
- **Modular Monolith**: A single deployment, single codebase, and single database (MongoDB) with strict internal boundaries. Components communicate via interfaces and event buses, not tight coupling.
- **Broker Agnosticism**: The core engine knows nothing about Upstox, Zerodha, or any specific broker. All broker interactions pass through strict Adapter interfaces.
- **Chart Agnosticism**: The frontend business logic does not depend on TradingView's proprietary data structures. An adapter layer translates generic payloads into chart-specific renders.
- **Non-Negotiable Design Rule (Frontend renders. Backend decides.)**: The frontend must never calculate ATH, Breakouts, Retests, Entries, Targets, Stop Losses, Signal Scores, or Risk. The frontend only visualizes backend decisions.

## 2. Tech Stack

### Backend
- **Framework**: FastAPI (Python 3.10+)
- **Database**: MongoDB (Motor async driver) + Beanie/Pydantic ODMs
- **Concurrency**: `asyncio` for high-throughput market data and socket streaming
- **Architecture**: Domain-Driven Design (DDD) with Service, Repository, and Adapter layers.

### Frontend
- **Framework**: Next.js 15 (App Router)
- **Language**: TypeScript
- **Styling**: TailwindCSS & ShadCN UI
- **State Management**: Zustand (Global state, Replay states)
- **Data Fetching**: TanStack Query (React Query)
- **Charting Engine**: TradingView Charting Library (Advanced, independent from widgets)

## 3. Backend Architecture & Features

The backend is organized into strict modules under `backend/app/`.

### A. Core Domain Models (`app/models/`)
The absolute center of the system. Everything revolves around these entities:
- **`Signal`**: Represents a trading opportunity generated by a strategy. Contains `signal_id`, `strategy_id`, `strategy_version`, `signal_score`, `confidence`, `entry`, `stop_loss`, `targets`, `risk_pct`, and a rigorous `SignalStatus` state machine (CREATED, ACTIVE, TRIGGERED, EXPIRED, CANCELLED).
- **`Trade`**: A unified model for both Paper and Live trades (differentiated by `trade_type`). It tracks `entry_price`, `qty`, `unrealized_pnl`, `realized_pnl`, and a rigorous `TradeStatus` state machine (OPEN, TP1_HIT, TP2_HIT, TP3_HIT, STOPPED_OUT, CLOSED).
- **`Note`**: User-generated textual observations mapped to a specific `symbol` or `signal_id`. Saved in the `annotations` collection.
- **`Watchlist`**: Tracks symbols of interest, including contextual data like All-Time Highs (ATH) and Volume ratios.
- **`Portfolio`**: Tracks capital, available cash, invested amount, and total PnL across PAPER or LIVE portfolios.
- **`StrategyConfig`**: Stores tunable parameters (e.g., ATH_DISTANCE, VOLUME_RATIO) so rules can be adjusted without code changes.
- **`AuditLog`**: Critical debugging trace capturing `entity_type`, `action`, and `payload`.
- **`Backtest`**: Stores historical performance runs, metrics, and equity curves.

### B. Scanners & Strategies (`app/scanners/`, `app/strategies/`)
Strict separation of concerns between *finding* stocks and *analyzing* them.
- **Scanners**: Scripts (like `chartink.py` or `ipo_base.py`) that filter the universe of thousands of stocks down to a small eligible list based on preliminary criteria (e.g., "Near ATH").
- **Strategy Registry**: A central dictionary mapping strategy strings to concrete classes implementing the `StrategyProtocol`.
- **Strategies**: Classes that implement `scan()`, `generate_signal()`, and `build_overlays()`. They contain the heavy mathematical rules.

### C. Services Layer (`app/services/`)
Decouples API routers from the database and business logic.
- **`SymbolMasterService`**: The global source of truth for instrument lookup.
- **`OverlayService`**: Reads a stock's active signals, queries the `StrategyRegistry` for specific overlay instructions, and returns a sanitized JSON array.
- **`RiskEngine`**: Receives signals from strategies and validates position sizing, SL ranges, RR ratios, and portfolio exposure before allowing them to become Approved Signals.
- **`NotificationService`**: Listens to the EventBus and routes messages to Telegram, Email, or Push notifications. Strategies never call Telegram directly.
- **`MarketSessionService`**: Evaluates Indian market hours (`is_market_open()`, `is_pre_market()`).

### D. Adapters: Brokers & Market Data (`app/brokers/`, `app/market_data/`)
- **`BrokerAdapter`**: Abstract base class enforcing methods like `place_order()`, `get_positions()`.
- **`MarketDataProvider`**: Abstract base class enforcing `search_symbols()`, `get_historical_candles()`.

### E. Infrastructure: Events, Schedulers & Caching (`app/events/`, `app/schedulers/`, `app/cache/`)
- **Event Bus**: Async in-memory Pub/Sub system (`events/bus.py`).
- **Schedulers**: Dedicated modules (`scanner_scheduler.py`, `alert_scheduler.py`) running cron jobs.
- **Cache**: Abstractions (`BaseCache`, `InMemoryCache`, `RedisCache`) to intercept expensive broker API calls.
- **Feature Flags**: Environment-driven toggles (`ENABLE_PAPER_TRADING`, `ENABLE_BACKTESTING`).

## 4. Frontend Architecture & Features

### A. The Trading Workstation (`/terminal/[symbol]/layout.tsx`)
A 5-pane professional UI layout designed for serious workflow:
1. **Watchlist Pane**: Displays active tracked stocks.
2. **Signal Details Pane**: Explains *why* the strategy triggered.
3. **Charting Pane**: Fullscreen TradingView canvas.
4. **Tool/Action Pane**: Contains Orders, Trades, Backtests, Notes, and Alerts tabs.

### B. Charting Engine Integration
- **`ChartAdapter.ts`**: Intermediary layer translating generic JSON models into TradingView's proprietary formats.
- **`datafeed.ts`**: Implements the `IDatafeedChartApi` by routing through `ChartAdapter`.
- **Data-Driven Overlays**: Queries `/api/v1/chart-overlays/{symbol}` and blindly paints them using primitive drawing tools.

## 5. Implementation Roadmap

Do not build everything simultaneously. Execute in strict phases:

1. **Phase 1**: TradingView, Market Data, Symbol Search, Terminal *(Current Sprint)*
2. **Phase 2**: Scanner, IPO ATH Strategy, Signal Generation, Overlays
3. **Phase 3**: Telegram, Paper Trading, Risk Engine
4. **Phase 4**: Backtesting, Portfolio Analytics, Replay Mode
5. **Phase 5**: Live Trading, Upstox Execution, Zerodha Execution
